RFC-003 — i18n：next-intl（App Router skeleton）

Status: Accepted
Owner: 子揚
Related: P2-M
Decision needed by: P2-M1 前
Last updated: 2026-02-26

⸻

1. 背景

目前前端使用 Next.js App Router（src/app）。我們要「先立 i18n 架構」，但不急著全站翻譯，避免未來全球化時大重構。

⸻

2. 目標 / 非目標

Goals
	•	建立最小 i18n skeleton：
	•	locale routing 結構（或至少 middleware-ready）
	•	messages namespace 結構
	•	date/number formatting 統一入口
	•	初期只跑 zh-TW 仍可正常運作（不增加不必要負擔）
	•	不做 repo-wide 文案翻譯 sweep

Non-goals
	•	不在此階段完成多語翻譯內容
	•	不在此階段導入 CMS 翻譯工作流

⸻

3. 方案比較

Option A（推薦）：next-intl（App Router）

優點：
	•	App Router 支援路徑清晰（middleware / provider / message loading） ￼
缺點：
	•	仍需少量 wiring

Option B：next-i18next

優點：
	•	生態熟
缺點：
	•	更常見於 Pages Router；App Router 需要更多 glue code，長期不一定更省

⸻

4. 決策

採用 Option A：next-intl，只做 skeleton（不做全站翻譯）。

⸻

5. Skeleton 規格（可落地且不擴散）

5.1 Locale Strategy（最小可行）

二選一（擇一即可，依你現況路由決定）：

A) 顯式 locale segment（未來擴語系最乾淨）
	•	/zh-TW/**（現在只放 zh-TW）
	•	未來加 /en/**

B) 不改 URL，但內部全走 messages（改動更小）
	•	URL 不帶 locale
	•	仍以 next-intl 管 messages 與 formatting
	•	未來要加 locale segment 時再做一次路由遷移（成本較高）

建議：如果你已準備 i18n 落地到 P2-M1 skeleton，優先選 A。

5.2 Messages Structure（必須 namespace）
	•	messages/zh-TW/common.json
	•	messages/zh-TW/decks.json
	•	messages/zh-TW/journal.json
	•	messages/zh-TW/settings.json
	•	禁止散亂 key：新增 key 必須落到 namespace 檔

5.3 Formatting（統一入口）
	•	禁止到處 new Date().toLocaleString()
	•	建議提供：formatDate() / formatNumber() / formatCurrency() 的單一模組入口（內部用 next-intl formatter）

5.4 Migration 原則（避免 sweep）
	•	不做 repo-wide sweep
	•	只在「新增功能 / 重點頁改 UI」時順手把該頁的硬編碼文案替換成 keys

⸻

6. DoD（最小落地）
	•	next-intl wiring（middleware / provider / message loading）完成且不破壞既有頁面 ￼
	•	zh-TW messages 具備 namespace 結構
	•	formatting 有統一入口
	•	1 個示範頁完成 key 使用（例如 settings 或 decks）

⸻

7. Rollback
	•	revert i18n skeleton commit：回到原本硬編碼文案
	•	保證不影響 API/DB
