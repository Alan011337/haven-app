RFC-002 — Haptic：PWA Vibration vs Capacitor（長期一致性）

Status: Accepted
Owner: 子揚
Related: P2-A（Haptic）, P2-G（Native readiness）
Decision needed by: Batch A11 後（RFC 完成再做 A12/A13）
Last updated: 2026-02-26

⸻

1. 背景

我們希望在「揭牌 / 提交成功 / 解鎖」提供觸覺回饋以增強儀式感。

PWA 常用 navigator.vibrate()（Vibration API），但 iOS Safari 多數情境不支援 → 體驗不一致（Android 有震、iOS 沒震）。 ￼

⸻

2. 目標 / 非目標

Goals
	•	提供一致的呼叫介面（hapticsService），由平台決定如何實作。
	•	Web 不支援時安靜 no-op（不 throw、不 console.error）。
	•	有設定可關閉（hapticsEnabled）與強度（light/medium），並有 cooldown 防止濫震。

Non-goals
	•	本階段不強制導入完整原生打包流程（除非決策採納 Capacitor）。
	•	不引入與 haptics 無關的原生插件鏈。

⸻

3. 方案比較

Option A（推薦，長期最好）：Capacitor Haptics-first + Web fallback

做法：
	•	統一由 hapticsService 提供 API
	•	Capacitor runtime → 用 Capacitor Haptics API
	•	否則（Web/PWA）→ 有支援才 vibrate，不支援就 no-op

優點：
	•	iOS/Android 一致、可控（原生級） ￼
	•	Web 端維持相容與安靜降級
缺點：
	•	導入 Capacitor 與打包流程會增加工程成本與維護面

Option B：PWA Vibration-only

優點：
	•	最省事、最少依賴
缺點：
	•	iOS/Safari 不支援 → 長期體驗斷裂與產品負債 ￼

⸻

4. 決策

採 Option A 的架構路線（先抽象後決策是否真的導入 Capacitor）：
	•	Batch A12：先落地 hapticsService（Web fallback + settings + no-op）
	•	Batch A13（可選）：批准後才做 Capacitor 整合

⸻

5. API 設計（建議）

hapticsService.trigger(event, strength?)
	•	events：
	•	reveal
	•	submit_success
	•	unlock
	•	strength：
	•	light | medium（可擴 heavy）
	•	cooldown：
	•	同事件 N ms 內不重複觸發（預設 250–500ms）

⸻

6. DoD（最小可驗收）
	•	settings：
	•	hapticsEnabled: boolean
	•	hapticStrength: "light" | "medium"
	•	事件掛鉤：reveal / submit_success / unlock
	•	Web：不支援時安靜 no-op
	•	若採 Capacitor：
	•	iOS/Android 觸發一致（至少 light/medium）
	•	Web 維持 no-op 或 vibrate fallback

⸻

7. 風險與緩解
	•	風險：導入 Capacitor 增加 build/CI 複雜度
緩解：先落地 service 抽象；Capacitor 另批次引入且可完全 revert
	•	風險：濫用 haptics 造成干擾
緩解：預設 strength=light + cooldown + 總開關

⸻

8. Rollback
	•	A12：revert 該 commit → 回到 no-op（或移除 service）
	•	A13：revert Capacitor integration commit → Web 維持 no-op/fallback
