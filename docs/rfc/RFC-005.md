RFC-005 — Scalability：Trigger metrics 先行（避免過度設計）

Status: Accepted
Owner: 子揚
Related: P2-B
Decision needed by: B1 前
Last updated: 2026-02-26

⸻

1. 背景

P2-B（partition / replica / WS Redis / cache / queue / tiered storage）屬於架構級改動。若沒有「觸發條件（Trigger metrics）」就先做，很容易：
	•	過度設計（複雜度先上升，收益不存在）
	•	引入不可逆操作成本（migration、維運、監控）

正確路徑：先定指標 + baseline，再依據數據決定要不要做下一步。

⸻

2. 決策：Trigger metrics（最小集合）

DB（Postgres）
	•	p95 query latency（按 endpoint 或主要 query 類型）
	•	slow query ratio（例如 > 200ms 的比例）
	•	connection pool saturation
	•	table size growth rate（journals / card_responses）
-（可選）replication lag（若引入 replica）

API
	•	RPS（按主要路由）
	•	error rate（5xx / timeout）
	•	p95 latency（端到端）

WebSocket
	•	concurrent connections
	•	broadcast frequency（每秒訊息量）
	•	message drop / backlog（若可觀測）

Queue（若引入）
	•	job success rate
	•	retry rate
	•	lag（排隊延遲）
	•	DLQ size

⸻

3. Baseline 規範（讓門檻可落地）
	•	baseline 必須同時記錄：
	•	數值（p50/p95、ratio）
	•	採樣期間（例如 7 天）
	•	負載情境（RPS、連線數、資料量）
	•	baseline 來源允許先從最小可行開始：
	•	DB logs + app logs + 簡易 dashboard
	•	pg_stat_statements（若已啟用） ￼

⸻

4. DoD（交付物）
	•	文件：docs/ops/scalability-triggers.md
	•	指標定義（如何量測）
	•	建議門檻（到什麼程度才需要做 Partition/Replica/Redis/Queue/ARCH）
	•	baseline 記錄方式與模板
	•	後續規則：
	•	每個 P2-B 子項都必須：先 RFC → 再 Spike → 再決策是否上線

⸻

5. 風險與緩解
	•	沒有監控就無法量測
→ 先用最小可行方式建立 baseline，再逐步完善
	•	門檻設太嚴/太鬆
→ 門檻可迭代，但必須先有 baseline；調整要留變更紀錄
