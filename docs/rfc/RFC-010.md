RFC-010 — Scalability Triggers（何時做擴展）

Status: Accepted
Owner: 子揚
Related: P2-B（擴展基石）、DATA-READ-01、CACHE-01、QUEUE-01
Decision needed by: 各項達門檻時再開專案 RFC / Spike
Last updated: 2026-02-23

⸻

1. 背景與問題

P2-B 目標是為千萬級、億級使用者預先鋪路，但「過早做」會增加複雜度與維護成本。必須先定義「何時需要做」的門檻，避免過度設計。

⸻

2. 目標 / 非目標

Goals
	•	定義 Scalability Trigger 指標與建議門檻（見下表）。
	•	達門檻時再啟動對應 RFC / Spike，不預先實作。
	•	觀測方式：可透過既有 health、APM、DB 監控取得或擴充。

Non-goals
	•	不在此 RFC 內實作任何擴展方案。
	•	不承諾具體閾值數字為最終值（可依實際負載調整）。

⸻

3. Trigger 指標與建議門檻

| 領域 | 指標 | 建議門檻（觸發討論） | 對應 P2-B 項 |
|------|------|----------------------|--------------|
| DB | p95 查詢延遲 | 持續 > 500ms（核心讀寫） | 分片 / Read Replica |
| DB | 單表行數 / 大小 | 單表 > 5M 行或 > 10GB | 分片準備 (journals / card_responses) |
| DB | 慢查詢比例 | 慢查詢佔比 > 5% | 索引 / 分片 / Read Replica |
| DB | 連線池飽和 | 使用率持續 > 80% | 連線優化 / Read Replica |
| WS | 單實例同時連線數 | 持續 > 2k | WebSocket 水平擴展 (Redis Pub/Sub) |
| WS | 廣播頻率 | 每秒廣播次數 > 100 | Redis Pub/Sub / 節流 |
| API | RPS（全站或關鍵路徑） | 持續 > 500 RPS | Read Replica / Cache / Queue |
| API | 錯誤率 | 5xx 或 timeout 率 > 1% | 穩定性優先，再談擴展 |

說明：以上為「何時開 RFC / Spike」的觸發條件，非自動上線條件。達門檻後應先寫 RFC（含 partition key、fallback、回滾），再進行 Spike 或小規模試點。

⸻

4. 各項擴展之 RFC 必含內容（達門檻時撰寫）

4.1 資料庫分片 / Partitioning（journals、card_responses）
	•	Partition key 選擇：created_at vs user_id，與查詢模式對齊。
	•	Partition 粒度：每月 / 每週；Postgres 官方提醒分太多 partition 會增加 planning time。
	•	索引策略、回滾策略（如何回單表、雙寫期間策略）。
	•	DoD：至少 1 張表 Spike + 基準測試（前後 p95）。

4.2 讀寫分離（Read Replica）
	•	哪些端點允許讀到稍舊資料（staleness 語意）。
	•	Replica 掛掉時 fallback 主庫。
	•	觀測 replication lag 與告警。

4.3 WebSocket 水平擴展（Redis Pub/Sub）
	•	Topic 設計：couple_id / user_id。
	•	At-least-once vs at-most-once 語意。
	•	失敗時降級策略（polling / 降級事件）。

⸻

5. 後續步驟

	•	在監控 / 儀表板中露出上述指標（若尚未有）。
	•	達門檻時：開對應 RFC，經共識後排 Spike 或試點。
	•	Read model / Cache / Queue（DATA-READ-01、CACHE-01、QUEUE-01）可另訂觸發條件或與本表對齊。
