RFC-007 — Read Replica Strategy（staleness + fallback + lag observability）

Status: Accepted
Owner: 子揚
Related: P2-B2（Read Replica）、P2-B4（Read model）、P2-I（BI/ETL）
Decision needed by: Sprint 2（若要開始做任何 replica / ETL）
Last updated: 2026-02-26

⸻

1. 背景與問題

為避免高頻讀取（歷史、feed、統計）拖累核心寫入（日記/抽卡），規劃讀寫分離。但最大風險是：
	•	staleness：哪些頁可容忍讀到稍舊資料？
	•	replica lag / replica down：如何觀測與自動 fallback？
	•	一致性錯覺：用戶剛寫完立刻去看卻看到舊資料

⸻

2. 目標 / 非目標

Goals
	•	定義哪些讀允許走 replica（容忍多少 staleness）
	•	replica lag 過大或故障 → 自動 fallback primary
	•	不改 API contract；只是內部資料來源路由
	•	觀測：replication lag、fallback rate、讀取延遲分位

Non-goals
	•	不在此 RFC 解決 multi-primary 或 multi-region
	•	不在早期把所有 read 都強制切 replica

⸻

3. Strong vs Eventual（明確清單）

允許走 replica（eventual）

可容忍 1–30 秒（依 baseline 調整）：
	•	歷史列表（journals history）
	•	feed（cursor pagination）
	•	趨勢/統計（尤其若 materialized/precomputed）
	•	BI/ETL

必須走 primary（strong）
	•	寫入後立刻回讀流程（create journal → show detail）
	•	權限/安全相關（authz、membership）
	•	金流/付費（未來）

⸻

4. 路由策略

Option A（推薦）：讀路由層 + fallback（strong/eventual 顯式標註）
	•	實作 read client（replica）+ write client（primary）
	•	每個 endpoint / repository function 標註 tier：strong | eventual
	•	replica lag > threshold 或 replica error → fallback primary（記錄事件 + 觸發 circuit breaker）

Option B：所有 read 都走 replica + 特例回 primary
	•	Pros：實作更少
	•	Cons：容易踩「立刻回讀」一致性雷；維護成本更高

採 Option A。

⸻

5. 觀測與門檻（需 baseline）

Lag 指標（至少一個）：
	•	replication lag seconds（依 infra/DB 指標）
	•	fallback count（因 lag/error 回 primary 的次數）
	•	replica query p95 / primary query p95

Threshold（先留空，等 baseline 填）：
	•	lag > X 秒 → 停止使用 replica（fallback + breaker）
	•	error rate > X → 暫停 replica 路由 N 分鐘

⸻

6. 最小落地（避免一次切太多）
	•	第一步只切「歷史列表」到 replica（feature flag 控制）
	•	觀測無誤、fallback 路徑穩定後，再擴其他 eventual pages

⸻

7. 驗收（DoD）
	•	strong vs eventual 的頁面/端點清單完整
	•	lag 觀測方案 + fallback/circuit breaker 設計
	•	最小落地：歷史列表走 replica（可 feature flag）
	•	replica off 時 app 行為不變（穩定回 primary）

⸻

8. Rollback
	•	feature flag off：所有讀回 primary
	•	revert 該 batch commit
