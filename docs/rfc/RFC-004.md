RFC-004 — Offline-first P0：Local queue + ack + replay（不碰 CRDT）

Status: Accepted
Owner: 子揚
Related: P2-F
Decision needed by: F1 前
Last updated: 2026-02-26

⸻

1. 背景與目標

在不穩網路（地鐵/飛機）情境下仍可使用：
	•	離線可操作（至少可寫入：日記 / 回答 / 核心互動）
	•	回線後自動補送（replay）
	•	本階段不碰 CRDT（避免架構爆炸）

⸻

2. 範圍定義（P0 / P1 / P2）

P0（本 RFC 核心）

Local queue + server ack + replay（先不碰 CRDT）
	•	IndexedDB op log（operation queue）
	•	operation_id（UUID）作為 idempotency key
	•	server 回傳 ack（acked / already_processed / rejected）
	•	client 回線後 replay + retry/backoff
	•	UI 顯示 pending / failed

P1（後續）

Conflict policy：LWW + semantic merge（要先定義 time source 與欄位策略）

P2（不在此階段落地）

CRDT（如 Yjs/Automerge）僅 RFC，不實作

⸻

3. 決策（P0 設計）

3.1 Operation Schema（建議）
	•	operation_id
	•	type：journal_create | journal_update | card_response_submit | ...
	•	payload：必要最小（避免把敏感/大量資料塞滿 queue）
	•	created_at_client
	•	retry_count
	•	last_error
	•	status：queued | inflight | acked | failed

3.2 Server 必要條件：Idempotency

對同一 operation_id 重送必須安全：
	•	不重複寫入
	•	回傳 deterministic ack（例如 already_processed）

⸻

4. 事件一致性原則（避免 UX 破碎）
	•	P0 不保證多端衝突合併，只保證「能補送」。
	•	UI 層：
	•	寫入成功前顯示 pending
	•	ack 後才視為真正成功
	•	failed 顯示可重試，且提供查看錯誤原因（可選）

⸻

5. DoD（可驗收）
	•	Offline 模式下：
	•	使用者可提交至少一類核心寫入（journal 或 card_response）
	•	UI 顯示 pending
	•	回線後：
	•	自動 replay
	•	成功後 pending 消失
	•	失敗顯示 failed，可重試
	•	retry/backoff：
	•	基本退避（1s, 2s, 4s… capped）
	•	不改 API contract 的前提下落地（若需要改，另開 RFC/決策點）

⸻

6. 測試（最小測試矩陣）
	•	手動：
	•	DevTools offline → 提交 → pending
	•	online → 自動補送 → ack → pending 消失
	•	模擬 server reject → failed → retry 後成功
	•	自動（可選）：
	•	queue serialize/deserialize 單元測試
	•	retry/backoff 邏輯測試

⸻

7. 風險與緩解
	•	多端同時寫入衝突
→ P0 不解，留給 P1（LWW/semantic merge）
	•	IndexedDB corruption / quota
→ 限制 queue 大小（上限 N 筆或容量）；超過提示使用者連線同步

⸻

8. Rollback
	•	revert Offline queue commit：恢復線上-only 行為
	•	保證不改 DB schema（若有變更必須另批次可回滾）
