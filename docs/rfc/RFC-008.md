RFC-008 — WebSocket Horizontal Scale via Redis Pub/Sub（topic + delivery + degrade）

Status: Accepted
Owner: 子揚
Related: P2-B3（WS Redis Pub/Sub）、P2-C（Time Capsule）、P2-D（Active Care）
Decision needed by: Sprint 2（若要改 WS Manager）
Last updated: 2026-02-26

⸻

1. 背景與問題

目前 WS 若是單機記憶體/單 instance，會受限於：
	•	同時連線數（單機資源）
	•	跨 instance 廣播（scale out 後事件無法 fan-out）

Redis Pub/Sub 可做跨 server 廣播，但它是 fire-and-forget / 不持久化：訂閱者不在線就會 miss 訊息，因此不適合作為可靠隊列。 ￼

⸻

2. 目標 / 非目標

Goals
	•	WS 廣播可跨多 instance（fan-out）
	•	定義 topic 命名與授權邊界（couple_id/user_id）
	•	明確宣告 delivery semantics（best-effort / at-most-once），並提供降級策略 ￼
	•	不改 API contract；只改 WS plumbing

Non-goals
	•	不把 Pub/Sub 當可靠訊息隊列（那是 QUEUE-01）
	•	不保證 WS 訊息不丟（改以 UI/HTTP 補救）

⸻

3. Delivery Semantics（必須寫死在文件）
	•	Redis Pub/Sub：訊息可能遺失（subscriber disconnected / network jitter） ￼
	•	因此 WS 事件應設計為：
	•	提示型（soft refresh / 你有新內容）
	•	或可被 HTTP 拉取補回（eventual consistency）

⸻

4. Topic 設計

建議 topic：
	•	couple:{couple_id}
	•	user:{user_id}
	•	system:broadcast（僅 admin/ops）

授權邏輯：
	•	server publish 前做 authz：只發到合法 topic
	•	client 訂閱 scope 由 server 決定（避免 client 任意訂閱）

⸻

5. 降級策略（非常重要）

當 Redis 不可用 / 延遲過大：
	•	WS 降級：
	•	polling（固定間隔拉狀態），或
	•	只保留本機 WS，不做跨 instance fan-out
	•	UI 原則：
	•	WS 不是唯一真相來源
	•	任何「必達」資訊必須能靠 HTTP/DB 查到

⸻

6. 觀測
	•	publish rate、subscribe client count
	•	Redis latency、error rate
	•	degrade / fallback 次數
-（可選）估算 drop（通常很難精準，但可用「應發事件 vs 客戶端 ack」近似）

⸻

7. 驗收（DoD）
	•	WS manager 支援 Redis Pub/Sub fan-out
	•	topic 命名與授權落地
	•	degrade path 可用（Redis off 不會讓 app 掛掉）
	•	文件清楚記錄：不可當可靠隊列 ￼

⸻

8. Rollback
	•	feature flag 關閉 Redis layer，回到單機 WS
	•	revert 該 batch commit
