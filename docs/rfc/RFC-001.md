RFC-001 — 引入 Framer Motion（限定範圍 + LazyMotion）

Status: Accepted
Owner: 子揚
Related: P2-A（Flip/Swipe/Glow）, P2-L（Motion System）, P2-K（A11y）
Decision needed by: Batch A3 前
Last updated: 2026-02-26

⸻

1. 背景與問題

我們需要「卡片儀式感」互動（Flip/Swipe/Glow），但目前前端未安裝 Framer Motion。引入後常見風險：
	•	Bundle 變大：若不限制 import 與使用範圍，容易造成全站依賴擴散。
	•	A11y：拖曳/手勢若成為唯一操作會違反 WCAG 2.2「Dragging Movements」的替代要求；點擊目標尺寸也要符合「Target Size (Minimum)」。 ￼
	•	Reduced motion：prefers-reduced-motion 情境需明確降級，避免眩暈與干擾。

⸻

2. 目標 / 非目標

Goals
	•	僅強化「卡片儀式」元件：TarotCard / DailyCard / Reveal flow（或等價元件）。
	•	prefers-reduced-motion 下可降級（淡入/切換，不做 3D 翻轉）。
	•	Swipe/Drag 一律提供非拖曳替代（上一張/下一張按鈕 + 鍵盤操作）。
	•	控制 bundle：必用 LazyMotion（避免整包載入）。
	•	可回滾：每個 batch 1 commit，可獨立 revert。

Non-goals
	•	不做 repo-wide 動效重構。
	•	不把 Framer Motion 當全站 animation framework（禁止擴散）。
	•	不替換既有全站 CSS transitions（除非在本 RFC 白名單範圍內）。

⸻

3. Scope Boundary（白名單 / 黑名單）

Allowed（允許使用 framer-motion 的路徑）

唯一長期允許路徑：
	•	frontend/src/components/features/card-ritual/**

短期過渡允許（最多 2 個 batch 內搬回 card-ritual）：
	•	frontend/src/components/ui/TarotCard.tsx
	•	frontend/src/features/deck-room/**（僅限儀式流程元件）

規則：若使用了過渡路徑，必須在後續 batch 新增搬遷 commit（仍 1 batch 1 commit）。

Forbidden（禁止 import framer-motion）
	•	frontend/src/app/**（layout/page 層禁止）
	•	frontend/src/components/ui/**（共用 UI 元件庫禁止）
	•	frontend/src/providers/**（禁止全站 provider）
	•	任何全站 wrapper / hook（除非另開 RFC）

⸻

4. 方案比較

Option A（推薦）：Framer Motion（嚴格 scope）+ LazyMotion

Pros：
	•	Flip/Swipe/Glow 互動細緻、可擴充
	•	LazyMotion 可降低初始載入負擔
Cons：
	•	仍增加依賴與 bundle
	•	需要守住護欄避免擴散

Option B：全用 CSS transition/animation

Pros：
	•	bundle 最小
Cons：
	•	Flip/Swipe 的狀態編排與手勢處理更容易堆 hack
	•	長期維護成本可能更高

⸻

5. 決策

採用 Option A，前提滿足 Guardrails：
	1.	motion 只允許在 components/features/card-ritual/**（或短期過渡白名單）
	2.	必用 LazyMotion（禁止全站 import motion 全家桶）
	3.	Drag/Swipe 必有按鈕與鍵盤替代（符合 WCAG 2.2 Dragging Movements） ￼
	4.	互動目標至少 24×24 CSS px（符合 WCAG 2.2 Target Size (Minimum)） ￼
	5.	必交付 bundle delta 紀錄（見 DoD）

⸻

6. Implementation Notes（落地規範）
	•	LazyMotion：只在 card-ritual root 做一次 features loader（例如 domAnimation）。
	•	Reduced motion：以 prefers-reduced-motion 或框架 hook 降級到淡入/切換。
	•	輸入替代：拖曳只是加分；「按鈕」與「鍵盤」是必須。
	•	事件一致性：Flip/Swipe 不得改變資料語意，只改視覺/互動。

⸻

7. 驗收（DoD）

Flip：
	•	Enter/Space 可觸發
	•	reduced-motion 下改淡入/切換（不得 3D 翻轉）

Swipe/Drag：
	•	有上一張/下一張按鈕
	•	鍵盤可操作
	•	拖曳不是唯一方式

Glow：
	•	可關閉（setting/flag）
	•	reduced-motion 下不得閃爍/呼吸高亮

Bundle：
	•	提交記錄「引入 framer-motion 後 build 輸出摘要」（例如：Next build report / chunk size 前後對比文字）

⸻

8. 測試
	•	cd frontend && npm run lint && npm run typecheck && npm run build
	•	手動：
	•	8 牌組：抽卡 / 揭牌 / 回答頁
	•	鍵盤操作（Enter/Space/方向鍵或 Tab 流）
	•	OS/Browser reduced-motion 開關

⸻

9. Rollback
	•	每個 batch 1 commit → 任一 batch 可 git revert <sha> 回到 CSS baseline。
	•	若依賴引入造成問題：revert 引入 framer-motion 的 commit，並移除相關引用（同樣 1 commit）。
