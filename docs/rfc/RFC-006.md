RFC-006 — Partitioning/Shard Readiness（先 Partition；Sharding 僅保留路線）

Status: Accepted
Owner: 子揚
Related: P2-B1（Partitioning）、P2-B4（Read model）、P2-B5（Cache）、P2-B6（Queue）、P2-J（Ops/Observability）
Decision needed by: Sprint 2 開始前（進 Spike 之前）
Last updated: 2026-02-26

⸻

1. 背景與問題

P2-B 想為「千萬/億級」預鋪路，原文提到對 journals / card_responses 依 created_at 或 user_id 做 Partitioning。設計不當會造成：
	•	partitions 太多 → planner overhead 上升、反而變慢（常見雷）
	•	索引/查詢複雜度上升（跨月/跨用戶查詢、排序、聚合）
	•	寫入/維護成本上升（vacuum、reindex、detach/attach、搬遷）

（先承認：Partition pruning 很強，但「分太細」同樣會帶來成本；因此必須先走 RFC-005 的 trigger + baseline，再做 spike。） ￼

⸻

2. 目標 / 非目標

Goals
	•	只有在達到 trigger 才啟動 Partitioning 落地（遵守 RFC-005）
	•	先做 1 表 Spike：最小可行驗證（不直接全表全量改）
	•	查詢語意不變：不改 API contract；外部行為維持
	•	可回滾：任一步都可 revert，或停留在「不啟用 partition routing」狀態

Non-goals
	•	不做真正多庫 Sharding（多 DB 分片）落地
	•	不在 Sprint 2 把 journals + card_responses 一次全改
	•	不為 partition 進行 repo-wide query layer 大重構（除非另寫 RFC）

⸻

3. Trigger metrics（啟動條件）

滿足任一條件才進入「Spike → 決策」：
	•	單表大小（含索引）> X GB（X 先由 baseline 填入）
	•	寫入 p95 latency > X ms 且與表增長呈強相關
	•	慢查詢比例 > X%（以 pg_stat_statements 觀測） ￼
	•	autovacuum / bloat / index bloat 對寫入造成顯著影響
	•	feed/歷史查詢掃描成本爆炸（seq scan 增多、buffer hit pattern 異常）

⸻

4. 方案比較（partition key）

Option A：按 created_at（時間分區：月/週）
	•	Pros：時間序列常見模式；與 cold archive（RFC-009）天然契合
	•	Cons：以 user_id 查歷史仍可能掃多 partitions（需 pruning + 索引策略）

Option B：按 user_id（hash/range）
	•	Pros：單用戶查詢更聚焦；與 couple_id 路徑可一致
	•	Cons：冷熱分離更麻煩；partition 數可能膨脹

Option C：Hybrid（time + subpartition by hash）
	•	Pros：兼顧歸檔與單用戶定位
	•	Cons：維運成本最高；不適合早期

建議：先選 Option A（created_at 月分區）做 Spike（最可控、最易回收）。

⸻

5. 落地策略（兩階段）

Phase 1 — Spike（只做 1 表）
	•	選表：journals 或 card_responses（挑「增長最快 + 查詢最痛」者）
	•	粒度：monthly partitions（先限制活躍 partitions 上限 N）
	•	索引：建立與現況等價的索引（避免因索引缺失造成假性退化）
	•	基準：對比同資料量下：
	•	query p50/p95
	•	write p50/p95
	•	planner time（EXPLAIN ANALYZE）

Phase 2 — 決策
	•	若改善顯著且維運可接受 → 才規劃第二張表
	•	若改善不顯著 → 停止並回收（不推進全量 partition）

⸻

6. 可觀測性（必做）
	•	pg_stat_statements：top queries、延遲分位 ￼
	•	partition pruning 命中：EXPLAIN 觀察掃描 partitions 數
	•	vacuum/bloat：監控 autovacuum 延遲與表膨脹

⸻

7. 驗收（DoD）
	•	RFC 完整：key/粒度/索引/回滾/觀測/測試寫清楚
	•	Spike 完成：至少 1 表，前後基準數據 + 結論
	•	不改 API contract
	•	1 commit 可回滾（測試環境 schema 可 drop/recreate）

⸻

8. Rollback
	•	Spike 層：在測試環境 revert schema changes / drop partitioned table，或停在「不切 routing」
	•	文件層：revert RFC/Spike 報告 commit
