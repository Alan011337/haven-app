RFC-009 — Tiered Storage (Hot/Cold) with S3（lifecycle + retrieval semantics）

Status: Accepted
Owner: 子揚
Related: P2-B7（ARCH-01 Tiered Storage）、P2-C（Archive/Memory Lane）、P2-I（BI/ETL）
Decision needed by: Sprint 2（若要真的做 cold archive）
Last updated: 2026-02-26

⸻

1. 背景與問題

P2-B7 提到「3 個月前數據封存至 S3 (Cold Tier)」。Tiered storage 牽涉到：
	•	查詢語意：用戶看 3 個月前資料是否需要等待？如何提示？
	•	索引/定位：如何知道哪筆在 cold、哪筆還在 hot？
	•	取回策略：on-demand rehydrate 還是預取？
	•	成本：storage class、lifecycle transition、retrieval cost

S3 Lifecycle rules 可將物件自動轉換到不同 storage class（例如 Standard-IA / Glacier 類）。 ￼

⸻

2. 目標 / 非目標

Goals
	•	定義：哪些資料進 cold（journals、attachments、card_responses？）
	•	定義：cold UX 語意（延遲、提示、預取）
	•	建立：封存索引（metadata）與可重跑封存 job
	•	不改 API contract（或最小改動且可回滾）

Non-goals
	•	不在此階段做 multi-region replication（那是 Ops road map）
	•	不一次把所有歷史資料搬走（先小規模、可回滾）

⸻

3. 資料分層策略（建議先從「附件」開始）

Hot（Postgres）
	•	最近 3 個月：完整可查、低延遲

Warm/Cold（S3）
	•	3 個月以前：以 object 存放（JSON/Parquet/zip，依需求）
	•	DB 只保留必要索引（metadata），例如：
	•	archive_index(user_id, date_range, s3_key, checksum, status, created_at)

建議：第一階段先封存「大物件/附件」→ 風險小、成本效益高；純文字 journals 可晚一點再做。

⸻

4. 查詢語意（UX）— 必須先決策

Option A（推薦）：先返回索引 + 延遲載入內容（async）
	•	Pros：UX 可控，不會卡死頁面
	•	Cons：需要 UI 支援「載入中/取回中」

Option B：server 同步 rehydrate 再回
	•	Pros：前端簡單
	•	Cons：延遲不可控、易 timeout、成本難控

採 Option A。

⸻

5. 封存流程（可重跑、可驗證）

週期：每日/每週 batch job（依量調整）
	1.	選出符合條件的資料範圍（例如 date_range）
	2.	生成 archive object（含 checksum、版本號、schema version）
	3.	上傳 S3（套 lifecycle rules）
	4.	寫入 archive_index（狀態：uploaded/verified）
5)（可選）刪除 hot 資料或只刪附件（刪除策略必須另開決策：資料保留/法務/復原）

S3 lifecycle transition 依 AWS 文件設計與限制（例如不同 storage class 的轉換條件） ￼

⸻

6. 風險與緩解
	•	取回慢
→ UI 提示 + 背景取回；避免同步阻塞（Option A）
	•	資料一致性
→ checksum + verify step；archive object 附 schema version
	•	成本不可控
→ 先限制範圍（只封存附件/非常舊資料），再擴大；lifecycle rules 做自動化控成本 ￼

⸻

7. 驗收（DoD）
	•	定義 cold 內容範圍與 UX 語意（延遲策略）
	•	archive_index 設計（不含敏感內容，只存定位與狀態）
	•	封存 job 可重跑 + 有 verify
	•	lifecycle 配置策略寫清楚（哪個 storage class、多久 transition） ￼

⸻

8. Rollback
	•	停止封存 job（最先做）
	•	保留 hot 資料不刪（最安全 rollback）
	•	revert 該 batch commit；archive objects 留在 S3 不影響主流程（但要記得成本治理）
