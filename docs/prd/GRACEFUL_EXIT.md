# PRD-RISK-01: The Graceful Exit（優雅退場）

## 目標

當伴侶關係結束或用戶選擇離開時，提供清晰、可預期且符合隱私的「優雅退場」產品流程，與 [LEGAL-01 數位分手協議](../legal/digital-breakup-protocol.md) 及隱私政策一致。

## 產品流程

1. **解除伴侶連結（Unpair）**
   - 任一方可於設定中主動解除與對方的伴侶連結。
   - 確認對話框明確說明：解除後雙方將無法再查看對方日記與共同卡片歷史；各自帳號與歷史資料保留。
   - 解除後：`partner_id` 設為 `NULL`（雙向由後端或單方觸發依產品決策）；API 與前端依 BOLA 與 `verify_active_partner_id` 確保不再存取對方資料。

2. **通知與狀態**
   - 可選：解除後對另一方發送「伴侶已解除連結」之 in-app 或 push 通知（依 trigger matrix `partner_unbound`），避免對方誤以為系統異常。
   - 前端：設定頁、首頁等處即時反映「未配對」狀態；引導至邀請伴侶或單人使用流程。

3. **資料與保留**
   - 解除連結不自動刪除任何日記或卡片紀錄；依 [資料刪除生命週期](https://github.com/.../DATA_RIGHTS.md) 與用戶主動「刪除帳號」請求處理。
   - 若用戶選擇「刪除帳號」，依資料權利政策執行匯出與刪除，並可引用 [LEGAL-02 數位遺產聯絡人](../legal/legacy-contact.md) 若已設定。

4. **冷靜期（可選）**
   - 產品可提供「提出解除後 N 日內可撤回」之選項，由產品與法務決定是否上線；若上線，需在條款與 UI 載明。

## 實作對照

- 後端：解除伴侶 API（如 `PATCH /api/users/me` 或專用 unpair 端點）將 `partner_id` 設為 `NULL`；雙向解除邏輯與 BOLA 防護。
- 前端：設定頁「解除伴侶」按鈕、確認文案、解除後狀態與引導。
- 通知：`partner_unbound` trigger 可選發送給對方；runbook 載明何時發送與節流。

## 與 LEGAL-01 之關係

本文件為產品流程說明；LEGAL-01 為數位分手協議（法務與條款對齊）。兩者需一致：解除後資料邊界、保留與刪除方式應在條款與隱私政策中載明。
